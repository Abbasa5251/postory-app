## 1. Product overview

**One-liner:** A team-based SaaS where small brand agencies generate social posts (copy + AI images), route them through internal and client approval, and schedule them to 6 platforms ‚Äî one workspace per client brand ‚Äî then prove the results with client-ready analytics.

**Positioning:** Not "cheaper Hootsuite." The pitch is the *closed loop small agencies actually run*: brief ‚Üí on-brand AI content ‚Üí internal review ‚Üí client approves from their phone ‚Üí auto-publish ‚Üí monthly client report. Competitors do pieces; we do the loop.

### 1.1 Personas
- **Agency owner** (better-auth `owner`) ‚Äî billing, seats, brands, everything.
- **Account manager** (`admin` / custom `approver`) ‚Äî reviews, approves, schedules, owns client relationships.
- **Content creator** (custom `creator`) ‚Äî drafts, generates, submits; sees only assigned brands.
- **Client stakeholder (external, no account)** ‚Äî approves posts and views reports via tokenized portal links.

### 1.2 Goals
- Routine post: brief ‚Üí scheduled in < 5 min.
- Zero posts published without required approvals (internal, and client where enabled).
- Publish reliability ‚â• 99.5% (beta), 99.9% (GA).
- Blended gross margin ‚â• 55% by month 6 post-GA (rising with Zernio band mix).

### 1.3 Non-goals (v1)
AI video generation (post-launch, D4) ¬∑ X/Twitter, Pinterest, Bluesky, Reddit, etc. (post-launch platforms) ¬∑ social listening ¬∑ paid ads ¬∑ unified DM inbox ¬∑ multi-step approval chains ¬∑ white-label portal domains ¬∑ native mobile apps ¬∑ free plan.

## 3. Architecture decision records

- [ ] **ADR-001 ‚Äî Our DB is the source of truth for posts.** Zernio post created only at schedule time; `zernio_post_id` stored; status driven by webhooks + 15-min reconciliation sweep. Zernio isolated behind a `publishing/` module.
- [ ] **ADR-002 (rev) ‚Äî Tenancy via better-auth organizations.** Org = agency. `organization`, `member`, `invitation`, `session` tables owned by better-auth migrations in our Postgres. Active org read from `session.activeOrganizationId`. **Every domain table carries `org_id` FK ‚Üí better-auth `organization.id`; every query flows through a data-access layer requiring `orgId`.** No webhook mirroring (Clerk's Epic-A4 deleted). This DAL is now the tenant-isolation *security boundary* ‚Äî see NFR + authz test suite. *(2026-07-15: better-auth org tables live in Neon; active-org set on sign-in/create/accept verified. DAL itself pending ‚Äî A5.)*
- [ ] **ADR-003 ‚Äî All generation & publishing async via Inngest.** Actions enqueue and return job ids; workers do the work; UI polls job status (2s while live). No AI/publish call in a request handler.
- [ ] **ADR-004 ‚Äî Direct Stripe.** Tier products (monthly + annual) + credit-pack one-off prices + `$9/mo` extra-account subscription item. Entitlements mirrored to `subscriptions` table; hot-path gates read our DB.
- [ ] **ADR-005 ‚Äî Append-only credit ledger.** Reserve before generation ‚Üí settle/refund after. Balance = materialized SUM per org.
- [ ] **ADR-006 ‚Äî Webhook ingestion.** Verify signature ‚Üí insert `webhook_events` (unique on provider event id) ‚Üí 200 fast ‚Üí process via Inngest. Applies to Stripe and Zernio (OpenRouter's Image API is synchronous request/response ‚Äî no webhook; revisit if async video jobs need one post-launch).
- [ ] **ADR-007 ‚Äî Media pipeline.** OpenRouter image output (base64) ‚Üí decode ‚Üí moderation ‚Üí R2 (`org/{orgId}/brand/{brandId}/‚Ä¶`) ‚Üí CDN; public URLs handed to Zernio at publish.
- [ ] **ADR-008 ‚Äî Client portal is token-based.** Signed, expiring, revocable tokens; scoped to approval sets *or* report views (same shell, two capabilities). All portal actions audited.
- [x] **ADR-009 (new; amended 2026-07-17, B1 grill; re-amended 2026-07-17, B3 grill ‚Äî R1 resolved; implemented 2026-07-18 in B3, PR #32 + #33) ‚Äî Brand ‚Üî Zernio profile is 1:1, profile provisioned lazily.** A brand owns exactly **one** Zernio profile, but the `zernio_profiles` row and the Zernio `POST /api/v1/profiles` call are created **on first account placement (B3), not at brand creation (B1)** ‚Äî a brand with zero connected accounts has no Zernio profile at all. This keeps brand creation a pure local mutation with no external dependency (Zernio being slow/down can't fail or slow a brand create). Invisible to users; `social_accounts.zernio_profile_id` records placement. Zero cost impact (billing is per account, profiles unlimited). **R1 resolution (B3 grill):** Zernio's own docs define a profile as *"a container ‚Ä¶ that groups connected accounts ‚Äî the tenant boundary"* with **no per-platform uniqueness** (accounts are freely relocatable via *move-account-to-profile*, and "multiple accounts per platform" is supported), so the earlier third-party "one-account-per-platform-per-profile" assumption is contradicted. The former overflow-profile mechanism (`profile_no` 2, 3, ‚Ä¶ auto-created on same-platform duplicates) is therefore **removed** and Brand‚Üîprofile collapses to 1:1 (a brand's one profile holds all its accounts, including multiples of the same platform). Docs are strong-but-not-conclusive on the exact same-platform-duplicate edge ‚Üí a one-line Zernio support confirmation is out (Epic B carry-over); the overflow codepath is not physically deleted from any lingering references until support replies, but no new code should assume >1 profile per brand.
- [ ] **ADR-010 (new) ‚Äî Trial enforcement server-side.** Trial state on the org row (`trial_ends_at`, `trial_state`). Inngest scheduled functions: T-3d nudge email ‚Üí expiry: flip to `read_only` ‚Üí +7d: disconnect Zernio accounts (stops account-day billing) + notify. Reactivation on subscribe restores connections list for one-click reconnect.
- [ ] **ADR-012 (new) ‚Äî All AI inference through OpenRouter (D11).** Text via chat completions (AI SDK OpenRouter provider); images via the dedicated Image API; video post-launch. One `OPENROUTER_API_KEY`. Model ids + prices live in `credit_rates`; per-model capabilities are read from OpenRouter's discovery endpoints, never hardcoded. `src/server/services/openrouter/` is the only module that knows the wire format ‚Äî preserving a direct-provider escape hatch. Accepted trade-off: one gateway for all AI; mitigated by OpenRouter's multi-provider routing/fallbacks and by not billing failed generations (simplifies credit refunds).
- [x] **ADR-011 (new) ‚Äî Auth security is ours now.** better-auth hardening checklist is a P0 deliverable: email verification required, password policy, rate-limited auth endpoints (Upstash), session revocation UI, `secondaryStorage` (Redis) for session lookups, secure cookie config, CSRF posture verified, audit login events. - [x] **ADR-013 (new) ‚Äî Server-action error contract.** Every mutation is authored through a hand-rolled `withAction(schema, permission, handler)` wrapper (not `next-safe-action`/`zsa` ‚Äî ¬ß7's pipeline is specific to our helpers) returning a typed `ActionResult` envelope. - [x] **ADR-014 (new) ‚Äî Interactive Zernio OAuth runs inline in request handlers.** A deliberate, narrow carve-out from ADR-003: the two interactive OAuth touchpoints (connect-initiation ‚Üí `authUrl` redirect, and the OAuth callback that persists the connected Social Account) run inline in request handlers ‚Äî like better-auth's own `/api/auth/*` routes ‚Äî because a sub-second user-blocking OAuth handshake is neither the generation nor the publishing workload ADR-003 governs. Lazy `POST /v1/profiles` provisioning and the callback's `GET /v1/accounts` read are covered by the same carve-out; all heavy/async Zernio work (health refresh, disconnection, reconciliation sweep, F3 webhooks) stays in Inngest. ## 5. Post lifecycle (v2 ‚Äî adds client review)

```
DRAFT ‚îÄ‚îÄsubmit‚îÄ‚îÄ‚ñ∂ IN_REVIEW ‚îÄ‚îÄrequest changes‚îÄ‚îÄ‚ñ∂ CHANGES_REQUESTED ‚îÄ‚îÄedit‚îÄ‚îÄ‚ñ∂ DRAFT
                     ‚îÇ internal approve
                     ‚ñº
            brand.requires_client_approval?
              no ‚îÄ‚îÄ‚ñ∂ APPROVED
              yes ‚îÄ‚ñ∂ CLIENT_REVIEW ‚îÄ‚îÄ client approves ‚îÄ‚îÄ‚ñ∂ APPROVED
                        ‚îÇ client requests changes
                        ‚ñº
                  CHANGES_REQUESTED (with client note)

APPROVED ‚îÄ‚îÄschedule‚îÄ‚îÄ‚ñ∂ SCHEDULED ‚îÄ‚îÄZernio‚îÄ‚îÄ‚ñ∂ PUBLISHING ‚îÄ‚îÄ‚ñ∂ PUBLISHED
                           ‚îÇunschedule                        ‚îÇ
                           ‚ñº                                  ‚ñº
                        APPROVED                       FAILED ‚îÄ‚îÄretry‚îÄ‚îÄ‚ñ∂ SCHEDULED
Any state ‚îÄ‚îÄ‚ñ∂ ARCHIVED
```

Rules (server-enforced, single module, exhaustively unit-tested):
- Approval binds to a `post_version` id; any edit after internal approval reverts to `DRAFT`.
- `CLIENT_REVIEW` entry auto-issues/refreshes a portal token and emails the client contact.
- Approving own post: org setting, default off.
- `PUBLISHING/PUBLISHED/FAILED` set only by Zernio webhook processor + reconciliation sweep.
- Every transition ‚Üí `audit_log`.
